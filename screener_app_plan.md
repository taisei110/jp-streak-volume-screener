# 連騰スクリーナー Streamlitアプリ化 計画書

## 0. この文書の使い方
- 本計画書(`screener_app_plan.md`)と既存スクリプト `jp_streak_volume_screener.py` を同じプロジェクトフォルダに置く。
- Claude Code に、文書末尾の「Claude Codeへのプロンプト」を貼り付けて着手させる。
- 1ステップごとに動作確認しながら進めること。

---

## 1. 目的
現在 `jp_streak_volume_screener.py` はコマンドライン実行で、パラメータを変えるたびにファイルを編集して再実行している。
これを Streamlit のローカルアプリにして、次の2つを可能にする。

1. 調整つまみ(閾値・モード等)を画面で動かすと、結果ランキングが即座に更新される。
2. 「データ取得」ボタンで任意のタイミングで株価を取得できる。ただし既に最新の場合は取得せず「取得済み」と表示する。

---

## 2. 前提・現状
- 技術スタック: Python / DuckDB / yfinance / pandas。アプリ層に Streamlit を追加する。
- 既存スクリプトの主な構成要素(これらを再利用する):
  - `get_universe()` … 銘柄リスト(コード→名前/市場)取得。tickers.csv 優先、無ければJPX Excel。
  - `fetch_to_db(codes, con)` … yfinanceで取得しDuckDB(`data/prices.duckdb`, テーブル`prices`)へ保存。
  - `screen(con)` … スクリーニング本体。結果DataFrameを返す。**現状モジュールのグローバル定数を直接参照している。**
  - `metric_cfg()` … 指標(出来高/売買代金)に応じた式・閾値・ラベルを返す。
  - `generate_html(res, data_period, run_time, stale_note)` … HTMLレポート生成。
  - `run_once()` / `main()` … CLI/daemonのエントリ。
- 主なパラメータ(モジュール冒頭の定数):
  `METRIC`, `STREAK_MIN`, `STREAK_TOP`, `BASE_WINDOW`, `VOL_MULT`, `VOL_MULT_SHORT`,
  `TURNOVER_VOL_MULT`, `TURNOVER_VOL_MULT_SHORT`, `VOL_MODE`, `MIN_VOLUME`, `MIN_TURNOVER`,
  `W_VOL`, `W_RISE`。
- データ取得(重い, 数分)とスクリーニング(軽い, 一瞬)が分離している。これがアプリ化に好都合な点。
  取得済みDBに対する `screen()` の再実行は一瞬なので、つまみ変更→即結果が成立する。

---

## 3. スコープ

### やること
- Streamlitアプリ `app.py` を**新規追加**する。
- `screen()` を引数で設定を受け取れるようにリファクタする(下記ステップ1)。
- データ鮮度判定ロジックを追加する。
- つまみUI / 結果表示 / 取得ボタン / 鮮度表示を実装する。

### やらないこと(スコープ外)
- 既存のCLI・daemon・バッチ(`setup.bat`等)・タスクスケジューラ自動取得の挙動変更。これらは一切壊さない。
- Discord通知やHTMLレポートの仕様変更(アプリからの呼び出しは任意機能としてのみ可)。
- クラウドへのデプロイ。ローカルの `streamlit run app.py` 前提でよい。

---

## 4. 維持すべき制約(壊してはいけないもの)
- `python jp_streak_volume_screener.py`(出来高版1回実行)、`--metric=turnover`、`--no-fetch`、`--daemon` の各CLI動作は**従来通り**動くこと。
- 自動取得(タスクスケジューラ/`run_daily.bat`/`--daemon`)はそのまま。アプリの取得ボタンは**それと独立した追加経路**であり、同じ `data/prices.duckdb` を共有する。
- DuckDBのスキーマ(`prices(code,date,open,close,volume)`)は変更しない。

---

## 5. 機能要件

### 5-1. サイドバー(調整つまみ)
以下をウィジェットで操作可能にする。初期値は現行のモジュール定数。
- 指標(`METRIC`): 「出来高 / 売買代金」のトグルまたはラジオ。
- `STREAK_TOP`(上位tierの連騰日数, 整数スライダー), `STREAK_MIN`(最小連騰日数)。
- 判定モード `VOL_MODE`: 「all / avg」のラジオ。
- 出来高版閾値: `VOL_MULT`, `VOL_MULT_SHORT`。売買代金版閾値: `TURNOVER_VOL_MULT`, `TURNOVER_VOL_MULT_SHORT`。
  → 選択中の指標に応じて該当する閾値スライダーのみ表示すると分かりやすい。
- `BASE_WINDOW`(基準中央値の期間)。
- 流動性フィルタ: `MIN_VOLUME`(株数), `MIN_TURNOVER`(円, 例: 0/1e8/...)。0で無効。
- ランキング重み: `W_VOL`, `W_RISE`(合計1になるよう一方から他方を算出してもよい)。

つまみを変更したら、**取得は行わず** `screen()` だけ再実行して結果表を更新する。

### 5-2. メイン画面(結果表示)
- 最上部に現在のデータ状態を常時表示: 「最終データ日: YYYY-MM-DD / 状態: 最新 or 要更新」。
- ランキング表: `rank, code, name, market, streak_len, tier, last_close, rise_pct, avg_ratio, min_ratio, score`。
  tier1/tier2の件数内訳も表示。
- 結果のCSVダウンロードボタン(任意で可)。
- 取得直後やデータが古い場合は鮮度の注意メッセージを出す(`generate_html`の`stale_note`相当)。

### 5-3. データ取得ボタン(本要望の中心)
押下時のフロー:
1. 直近営業日(下記6章)を求め、DBの最新日と比較する。
2. **既に最新の場合**(DB最新日 >= 直近営業日): 取得を実行せず、`st.info` 等で
   「既に最新のデータです(最終取得日: YYYY-MM-DD)」と表示して終了。
3. **古い場合**: 取得中のスピナー/進捗を出しつつ `fetch_to_db()` を実行し、完了後
   「N銘柄を更新しました(最終取得日: YYYY-MM-DD)」と表示。取得後は結果表も更新する。
- 取得はこのボタンを押したときだけ走る。アプリ起動やつまみ変更では絶対に取得しない。

---

## 6. 重要ロジック: データ鮮度判定
「既に最新かどうか」の判定基準を明確にする。曖昧だと無駄取得や誤判定が出る。

- `latest_trading_day(now)` を実装する。直近の「市場が動いた営業日」を返す。
  - 土日を除外。`jpholiday` が利用可能なら日本の祝日も除外(未導入なら曜日のみで判定し、その旨ログ)。
  - 当日が営業日でも、yfinanceは大引け直後に当日終値を反映しないことがある。
    そこで「現在時刻がその日の market close + 反映待ち(例: 18:00)以降」なら当日を対象、
    それ未満なら直前営業日を対象とする。閾値時刻は定数化(例 `FRESH_CUTOFF_HOUR = 18`)。
- `is_data_fresh(con)` = `DB の MAX(date) >= latest_trading_day(now)`。
- この関数は取得ボタンの判定と、画面上部の状態表示の両方で使う。
- エッジケース(DBが空, 全銘柄が休止等)では「要更新」側に倒して安全にする。

## 7. 実装ステップ(この順で、各ステップ後に動作確認)

**Step 1: 設定の引数化(リファクタ)**
- パラメータ群を1つの設定オブジェクト(`@dataclass ScreenConfig` 推奨, またはdict)にまとめる。
- `screen(con, cfg)` と `metric_cfg(cfg)` が設定を引数で受け取る形に変更。
- 既存の `run_once()`/`main()` は、モジュール定数から `cfg` を構築して渡すようにし、**CLI挙動は不変**に保つ。
- 確認: `python jp_streak_volume_screener.py --no-fetch` と `--no-fetch --metric=turnover` が
  リファクタ前と同じ結果を返すこと(件数・上位数件を目視で回帰確認)。

**Step 2: コア関数の共有可能化**
- `app.py` から `get_universe / fetch_to_db / screen / ScreenConfig / generate_html / 鮮度関数` を
  import できるようにする(同一ファイルからのimportで可。必要なら薄い `screener_core.py` への分離も可)。

**Step 3: 鮮度判定の実装**
- 6章の `latest_trading_day` と `is_data_fresh` を実装し、単体で動作確認。

**Step 4: app.py(UI)の実装**
- サイドバー(5-1)、メイン表示(5-2)、取得ボタン(5-3)を実装。

**Step 5: キャッシュ戦略**
- DuckDB接続と銘柄ユニバースは `st.session_state` で保持し再生成しない。
- `screen(con, cfg)` の結果は cfg をキーにキャッシュ(`st.cache_data` 等)し、つまみ変更時のみ再計算。
- 取得は明示ボタンのみ。取得後はキャッシュを無効化して再描画。

**Step 6: 仕上げ・確認**
- 受け入れ基準(8章)を満たすか確認。READMEに `streamlit run app.py` の起動手順を追記。

## 8. 受け入れ基準
- [ ] 既存CLI(`通常 / --metric=turnover / --no-fetch / --daemon`)がリファクタ後も従来通り動作する。
- [ ] アプリ起動時、取得は走らず、既存DBの内容で即ランキングが表示される。
- [ ] サイドバーのつまみを変えると、取得なしで結果表が即更新される。
- [ ] 指標(出来高/売買代金)の切り替えで、閾値UIと結果が連動して変わる。
- [ ] 取得ボタン: データが古いときだけ取得し、最新のときは「既に最新(最終取得日)」と表示して取得しない。
- [ ] 画面上部に最終データ日と鮮度状態が常時表示される。
- [ ] `data/prices.duckdb` を自動取得と共有し、スキーマを壊さない。

## 9. 自律実行の運用方針
本作業は各ステップで人間の承認を待たず、Claude Codeが自律的に Step 1→6 を進める。
ただし「自己検証ゲート」と「停止条件(赤信号)」を設け、緑なら自走・赤なら停止とする。

### 9-1. 自己検証ゲート(各ステップ完了時に必ず実行)
- Step 1 後: 保存済みベースライン CSV(`baseline_volume.csv` / `baseline_turnover.csv`)と
  リファクタ後の再実行結果を**スクリプトで自動比較**し、銘柄・順位・score の完全一致を機械判定する。
  一致(緑)なら Step 2 へ。不一致(赤)なら停止。
- Step 3 後: 鮮度判定関数を、複数の日付・時刻・祝日ケースで単体検証し、想定通りなら次へ。
- Step 4-5 後: アプリが起動し、起動時に取得が走らないこと、つまみ変更で取得なしに結果が更新されること、
  取得ボタンが鮮度に応じて分岐することを確認。
- 各ゲートの結果(緑/赤と根拠)を簡潔にログに残す。

### 9-2. 停止条件(赤信号 = 自動で進めず即停止して報告)
1. リファクタ後の結果がベースライン(12銘柄 / 7銘柄)と一致しない。
2. 受け入れ基準(8章)のいずれかを満たせない。
3. 既存の CLI・daemon・自動取得を壊さずには進められない。
4. DuckDB スキーマの変更が必要になった。
5. ファイル削除・git push・環境変数の永続変更など、不可逆/環境影響のある操作が必要。
- 依存追加(streamlit 等の pip install)は実行してよい。ただし何を入れたか報告する。
- ベースライン比較用ファイルは全工程完了まで削除しない。

### 9-3. 完了報告
全 Step 完了後に、(a) 受け入れ基準の達成状況、(b) 変更/追加したファイル一覧、
(c) `streamlit run app.py` の起動手順、をまとめて報告する。
